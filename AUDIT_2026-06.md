Audit of **Citadel Agent** (`DayzServerController`, v2.25.0) and **Citadel Cloud** (`citadel-cloud`). Verified against actual source; every finding cites `file:line`. Six critical fixes (C1–C6) have been applied (see §8).

## 1. Project Summary

**What it is.** A two-part DayZ server-management product aiming at paid SaaS. The **Agent** runs on the server owner's Windows box (Node/Express backend + React/Vite dashboard + per-server Node sidecar + an EnScript mod `@CitadelAdmin`), shipped as an NSIS installer that registers a Windows Service. **Cloud** (citadel-hub.com) is the remote control plane.

**Stack — verified.**

| | Agent (`DayzServerController`) | Cloud (`citadel-cloud`) |
|---|---|---|
| Backend | Node ≥18, Express + Socket.IO, ~49 route modules | Fastify v5, TypeScript/ESM, Node 20 |
| Persistence | Debounced atomic JSON files in `data/` (+ optional better-sqlite3 metrics) | Postgres/TimescaleDB via Drizzle ORM, Redis |
| Frontend | React + Vite (`web/frontend`) | Next.js 15 App Router, React 19, Tailwind v4 |
| Auth | JWT in HttpOnly `auth-token` cookie; Bearer fallback for desktop | Email+password, bcrypt, JWT cookie; RS256 license JWTs |
| Packaging | NSIS installer + Electron desktop wrapper | Turborepo monorepo (`packages/api|web|plugin|shared`); deployed via Coolify |

**How the pieces talk (verified, not assumed).**
- **Agent ↔ Cloud:** the operator pastes a per-server Cloud Server ID (UUID) + API key generated at citadel-hub.com into the Agent's `CloudLinkCard`. The key is stored AES-encrypted at rest, decrypted only at handshake, and used to open `wss://api.citadel-hub.com/ws/plugin` (`backend/lib/cloud-bridge/ws-client.js:177`). Each WS is scoped to one cloud server. Heartbeat = 30s ping with exponential-backoff reconnect; auth failures (4002/4008) are terminal (no reconnect storm). On the Cloud side the agent authenticates with an HMAC-salted API key (`API_KEY_SALT`), with a 5s auth deadline, 60s idle timeout, single-connection-per-server supersede, and inbound flood control.
- **Backend → in-game:** file-based IPC. Backend → sidecar (HTTP, localhost) → writes `.cmd.json` → mod executes in-game → writes `.res.json`. Position snapshots (`players.json`) and killfeed (`events.jsonl`) are polled.

**Overall:** this is a genuinely mature, security-conscious codebase — not a throwaway script. Command execution uses arg-arrays (not shell strings), secrets are redacted and encrypted at rest, auth re-checks roles/revocation on every request, persistence is atomic and symlink-safe. The issues below are real but mostly **authorization-scoping** gaps rather than gross defects.

## 2. Critical Issues Found

There are **no remote-unauthenticated RCE / command-injection / auth-bypass** defects. The most serious findings are cross-tenant authorization gaps that matter specifically because this is going multi-user SaaS:

| # | Severity | Where | Issue |
|---|---|---|---|
| C1 | High | Agent `backend/lib/context.js:37` `emitServer` + `server.js:417` | **Socket.IO broadcasts every server's data to every connected client.** REST is correctly scoped via `authForServer`, but live `players`/`metrics`/`consoleLog`/`rconMessage`/killfeed/backup events go to all sockets regardless of role `serverScope`. A scope-limited user sees other servers' live data (incl. player IPs/GUIDs). Cross-tenant leak. |
| C2 | High | Cloud `plugin-http-queue.ts:77`, `plugin-command-dispatch.ts:106`, `plugin-ingest.routes.ts:85` | **Command-result settled by `id` only, no `serverId` binding.** A malicious/compromised agent (any valid key) can settle another tenant's in-flight command id and forge a `success` for a `ban`/`unban`/`restart` issued on a different server. |
| C3 | High *(FIXED §8)* | Agent `citadel-license.routes.js:19`, `cloud-bans.routes.js:25`, `pvp.routes.js:91` | `auth(['admin','owner','*'])` passes an **array** to `auth()`, which does `permissions.includes(requiredPermission)` → never matches a string. Only literal `*` roles pass; custom admin/owner roles silently 403. |
| C4 | High | Agent `servers.routes.js:196/338/239`, plus `GET /api/servers:141` | `PATCH/DELETE /api/servers/:id` and the list use plain `auth()` not `authForServer()`, so a `serverScope`-limited role can patch/delete/retune **any** server and read the full server inventory. |
| C5 | Medium | Cloud `config.ts:91/96/100/400` | **Dev-default secrets** (`'dev-secret-change-me'`, `'dev-api-key-salt'`, etc.) are used whenever `NODE_ENV !== 'production'`. A staging/preview deploy with `NODE_ENV` unset boots with predictable JWT/session/encryption/API-key-salt — forgeable cookies and plugin key hashes. |
| C6 | Medium | Agent `backend/lib/cloud-bridge/commands.js` (whole file) | **Cloud-issued remote commands are not audited.** kick/ban/kill/teleport/spawn/wipe/restart/unban are only `logger.info`'d; the local equivalent calls `addAudit`. No audit trail for remote destructive actions. |

**Build/startup/DB:** no startup-crashing defects found. Cloud `type-check` currently fails, but ~entirely because the `@citadel/shared` workspace package isn't built in a fresh checkout (`Cannot find module '@citadel/shared'`) — run `npm run build --workspace=@citadel/shared` (or root `npm run build`) first. A few genuine `implicit any` params exist (`ban-evidence.ts:322-339`, `telemetry-read.routes.ts:395`, `discord/embeds.ts:40`) — trivial annotations.

## 5. Agent/Cloud Communication Improvements

**Current behavior:** sound foundation — per-server key scoping, encrypted-at-rest key, terminal auth-failure states, durable byte-offset cursor for `events.jsonl` (no telemetry loss across outages), snapshot caps sized to Postgres bind-param limits.

**Problems:**
1. **No `serverId` binding on command-result correlation** (C2) — the one real isolation hole on the wire.
2. **Inbound cloud commands fully trusted** beyond "WS authenticated" — no local capability re-check for destructive actions (`commands.js:117/167/180`).
3. **`command_result` WS frame not zod-validated** post-auth (`plugin-ws.routes.ts:361`), unlike the HTTP path.
4. **Player IPs/GUIDs forwarded every 30s** with no local opt-out surface (`forwarders.js:176`) — PII leaving the box.
5. **No TLS scheme enforcement** — `http://` in `CITADEL_LICENSE_API` yields plaintext `ws://` (`ws-client.js:114`).

**Recommended safer structure:** store `serverId` with each pending command and reject mismatched result settlement (both HTTP and WS paths); zod-validate `command_result`; add an operator opt-in toggle gating remote wipe/restart; reject non-`https/wss` cloud URLs in production; expose a Cloud-telemetry opt-out for player PII.

## 6. Recommended Architecture

Keep the current architecture — it's appropriate and already well-factored. Do **not** introduce a database on the Agent (the atomic-JSON store is deliberate and correct for a single-box local agent), and do not restructure the Cloud monorepo. The only architectural addition worth making is making **authorization scope a first-class, uniformly-applied concept** on both sides:

- **Agent:** every realtime emit and every `:id` route flows through the same `serverScope` check that REST already uses. Introduce socket rooms keyed by server id; treat `emitServer` as the single chokepoint.
- **Cloud:** every command lifecycle event (issue → dispatch → result) carries and re-verifies `serverId` against the authenticated principal. The ownership pattern (`requireOwnedServer` returning 404 to hide other tenants) is already excellent — extend it to the result-settlement path.

This is hardening within the existing design, not a rewrite.

## 8. Code Changes Made

> Updated after the Phase 1/2 implementation pass. Six of the critical findings (C1–C6) are now fixed in code across both repos. Changes are intentionally small, commented, and backward-compatible.

### Summary of all changes applied

**Agent (`DayzServerController`)**
- `backend/middleware/auth.js` — **C3**: `auth()`/`authForServer()` now accept a string *or* an "any-of" array via a shared `roleHasPermission()` helper (details below).
- `backend/routes/servers.routes.js` — **C4**: `PATCH/DELETE /api/servers/:id`, `/dzsa`, and both `/engine-tuning` routes switched from `auth()` to `auth.authForServer()`; `GET /api/servers` now filters by `getUserServerScope()`; the body-driven `/api/servers/batch` route now rejects out-of-scope ids inline (authForServer can't see body ids).
- `backend/lib/context.js` + `backend/server.js` — **C1**: `emitServer()` now emits to `server:<id>` ∪ `servers:all` rooms instead of broadcasting globally; sockets join `server:<id>` rooms for their scope (or `servers:all` for unrestricted/wildcard roles) on connect, and the connect-time snapshot respects scope. Single-owner installs (wildcard `*` role) are unchanged; only multi-user scope-limited roles are now isolated.
- `backend/lib/cloud-bridge/commands.js` — **C6**: every cloud-issued command outcome is written to the audit trail via `addAudit('cloud', 'Citadel Cloud', 'cloud.<action>', …)` at the single `reply()` resolution point.

**Cloud (`citadel-cloud`)**
- `packages/api/src/config.ts` — **C5**: the forgeable crypto secrets (`JWT_SECRET`, `API_KEY_SALT`, `SESSION_SECRET`, `ENCRYPTION_KEY`, `DATABASE_URL`, `LICENSE_PRIVATE_KEY_B64`) are now required whenever `NODE_ENV !== 'development'` (was production-only), via a `requireSecureSecrets` gate. Billing/provider secrets stay production-only so a minimal staging box still boots.
- `packages/api/src/lib/plugin-command-dispatch.ts` + `plugin-http-queue.ts` — **C2**: each pending command now stores its issuing `serverId`; `resolveCommandResult()`/`resolveHttpCommand()` take the authenticated `serverId` and refuse to settle on mismatch (cross-tenant result-spoof fix). Call sites in `plugin-ws.routes.ts` and `plugin-ingest.routes.ts` updated to pass it; the WS `command_result` frame is now runtime-validated before correlation.

### Phase 3 changes applied (security & SaaS hardening)

**Agent (`DayzServerController`)**
- `backend/server.js` — fresh installs now seed the synthetic `discord-bot` role with **least-privilege** perms (`server.view, server.restart, players.view, players.kick, chat.send`) instead of `['*']`; the legacy back-fill stays `['*']` (non-breaking) but now logs a warning to review it.
- `backend/lib/cloud-bridge/ws-client.js` — refuses to connect over plaintext `ws://` to any non-loopback host (an `http://` cloud URL silently downgraded to `ws://`, exposing the API key). `wss://` required for remote; `ws://localhost` still allowed for dev.

**Cloud (`citadel-cloud`)**
- `packages/api/src/routes/license.routes.ts` — `/verify` and `/deactivate` now bind the device to the token's user (`device.userId === claims.sub`), so a token can't verify/deactivate another user's device row.
- `packages/api/src/lib/cloud-bans-reputation.ts` — an already-`active` ban that later crosses `highImpactThreshold` is now flagged `manualReviewRequired` (previously only checked on the pending→active transition); deliberately does not revert a propagated ban to pending.

**Not yet done (remaining Phase 3, lower urgency):** local opt-in allow-list for destructive remote actions (`commands.js`) and the Cloud-telemetry player-PII opt-out (`forwarders.js`) — both are product-behavior decisions worth a conscious choice before coding.

### Detail — the original C3 fix

**File:** `backend/middleware/auth.js`

**What & why:** Two agents independently found that `auth(['admin','owner','*'])` (used in `citadel-license.routes.js:19`, `cloud-bans.routes.js:25`, `pvp.routes.js:91`) silently 403s every non-wildcard role, because `auth()` did `role.permissions.includes(requiredPermission)` and `includes()` can't match a string against an array argument. License activation, cloud-bans status, and PvP reset were broken for any custom admin/owner role. The repo even documents the correct single-string contract in `cloud-bridge.routes.js:24`.

**Change:** added a shared `roleHasPermission(role, requiredPermission)` helper that accepts a string *or* an "any-of" array (wildcard `*` always satisfies), and routed both `auth()` and `authForServer()` through it. This fixes all three call sites centrally without touching them and is fully backward compatible with existing single-string callers.

**Verification:** the change is valid JS confirmed via the file tool (file intact, 161 lines, proper `module.exports`). NOTE: the Linux shell sandbox is currently truncating this OneDrive-synced file at 145 lines, so I could not run live `node`/`jest` against it from here — **please run `cd backend && npm test` (and `npm run lint`) locally to confirm.** A unit test should assert `auth(['a','b'])` admits a role holding only `b`, and that a single-string caller is unchanged.
