CRITICAL FIXES APPLIED:

1. ATOMIC MOD INSTALLATION
   File: backend/lib/mod-manager.js
   Issue: Mod corruption on mid-copy failure
   Fix: Stage-then-swap pattern with atomic rename + backup restore
   Impact: CRITICAL - Prevents server instability from corrupted mods
   
2. POST-INSTALLATION VALIDATION
   File: backend/lib/mod-manager.js
   Issue: No validation after install, corrupted mods slip through
   Fix: Multi-point validation (signatures, directory health, file count)
   Impact: MEDIUM - Catches incomplete/corrupted installations
   
3. WORKSHOP API RATE LIMITING
   File: backend/lib/workshop.js
   Issue: API throttling/blacklisting from excessive requests
   Fix: Token bucket algorithm (10/min, 100/hour) + exponential backoff
   Impact: CRITICAL - Prevents API blacklisting
   
4. MOD CACHE TTL & INVALIDATION
   File: backend/lib/mod-cache.js
   Issue: Stale mods served indefinitely
   Fix: 30-min TTL with Steam time_updated comparison + LRU eviction
   Impact: HIGH - Ensures fresh mods, bounded cache size
   
5. DISK SPACE MANAGEMENT
   File: backend/lib/mod-cache.js
   Issue: Cache could consume entire disk
   Fix: Disk space checks, low-space warnings, automatic LRU eviction
   Impact: MEDIUM - Prevents disk exhaustion
   
6. AUTO-UPDATER STATE ATOMICITY
   File: backend/lib/auto-updater.js
   Issue: Crash between state transitions corrupts update state
   Fix: Write-ahead journal with crash recovery
   Impact: CRITICAL - Prevents lost updates and corrupted state
   
7. AUTO-ROLLBACK ON START FAILURE
   File: backend/lib/auto-updater.js
   Issue: Server crashes after update, no recovery
   Fix: 60s post-start verification + automatic backup restore
   Impact: CRITICAL - Prevents unrecoverable server states
   
8. BACKUP FAILURE ABORTS UPDATE
   File: backend/lib/auto-updater.js
   Issue: Update proceeds despite backup failure
   Fix: Pre-update backup validation, abort if fails
   Impact: CRITICAL - Prevents data loss scenarios

KNOWN LIMITATIONS:

1. Rate limiting is per-process (not distributed across servers)
   - If running multiple instances, implement shared rate limiter
   
2. Disk space check relies on fs.statfsSync (Node 18.17+)
   - Older Node versions may skip disk checks (non-critical)
   
3. State journal requires writable C:\Citadel\data\ directory
   - Must be writable by service account
   
4. Auto-rollback requires pre-update backup enabled
   - If no backup backup exists, rollback cannot occur (logged)

ROLLBACK PLAN:

If issues found post-deployment:

1. Atomic Mod Install Issues:
   - Manual cleanup: Remove _staging_* and _backup_* folders in mod directory
   - Restore mod from backup or re-download from Steam
   
2. Cache Issues:
   - Clear entire cache: rm -r C:\Citadel\cache\mods\
   - Cache will auto-repopulate on next mod use
   
3. Rate Limiting Issues:
   - Reduce perMinute/perHour limits in workshop.js
   - Restart backend service
   
4. State Journal Issues:
   - Remove .journal.json files: rm C:\Citadel\data\state-journals\*.journal.json
   - Restart backend service
   - States will return to idle on restart

5. Full Rollback:
   - Revert files to previous version
   - Clear cache: rm -r C:\Citadel\cache\mods\
   - Clear journal: rm -r C:\Citadel\data\state-journals\
   - Restart backend service
